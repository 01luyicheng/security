### 9.1 暴力破解测试

#### 9.1.1 小米路由器登录测试

**测试工具：** Hydra 9.6  
**字典：** rockyou.txt (14,344,399条密码)  
**测试目标：** 192.168.20.23 (小米路由器)

**测试结果：**
- 使用admin用户名进行HTTP POST表单暴力破解
- 测试URL: `/cgi-bin/luci/api/xqsystem/login`
- **发现4个有效密码：**
  - 12345
  - 123456
  - 123456789
  - password

**验证测试：**
- 尝试使用发现的密码登录，但返回错误："Invalid token"
- 原因：小米路由器需要额外的token认证机制
- 结论：虽然发现了弱密码，但由于token保护，无法直接登录

#### 9.1.2 摄像头服务测试

- 摄像头设备配置了较严格的访问控制
- 所有常用服务端口被过滤

## 附录A：网络出口拓扑与中间设备分析

> **分析时间：** 2026-06-10 ~ 2026-06-11  
> **分析手段：** 路由追踪、HTTP 头指纹分析、端口扫描、DNS 解析对比  
> **测试节点：** 192.168.20.29（有线接入点）

### A.1 确认的网络出口拓扑

通过 TTL 分析与多节点探测，确认学校网络存在**三层 NAT/路由架构**：

```
无线用户 (10.5.190.x / 10.5.191.x)
        │
        ▼
锐捷 AP / 接入交换机 (192.168.101.x 管理网)
        │
        ▼
学校核心网关 (192.168.20.1) — 锐捷 ReyeeOS
        │
        ▼
中国电信智能网关 (192.168.1.1) — 光猫拨号
        │
        ▼
     公网出口
```

**关键证据：**
- `ping 192.168.1.1` TTL=62，说明从本机出发经过 **2 跳**到达光猫（本机 → 192.168.20.1 → 192.168.1.1）。
- 192.168.1.1 网页标题为 **"中国电信智能网关"**，使用 OpenWrt/LuCI 框架，确认是运营商光猫。
- 学校网关 192.168.20.1 与光猫 192.168.1.1 为不同设备（HTTP Server 指纹、Etag、响应头均不同）。

### A.2 透明代理/网络安全设备证据

在对学校网络进行代理合规性测试时，发现出校流量**存在透明代理或上网行为管理设备**：

| 证据类型 | 具体发现 | 说明 |
|---------|---------|------|
| HTTP 头指纹 | `Proxy-Connection: keep-alive` | 访问阿里云、腾讯云、中科大镜像、百度、光猫自身均出现此头，源站本身不会返回 |
| Traceroute 隐藏 | ICMP/UDP 全星号；TCP 直接到目标 | 中间设备过滤了 ICMP TTL 超时包，且对 TCP SYN 做代理/转发 |
| 国外 IP 延迟 | `ping 1.1.1.1` 约 220ms | 延迟偏高，符合"多一跳"或汇聚到统一安全出口的特征 |
| DNS 劫持 | `icube-api.bytedance.net` → `10.8.7.221` | 学校 DNS 对特定域名做内网重定向，可能指向本地缓存或审计节点 |

**SSL 证书状态：** 对阿里云、百度等站点检查证书链，均为正常 GlobalSign 签发，**未发现 SSL 中间人替换**。

### A.3 扩展网段设备指纹对比

进一步探测了学校各网段的关键节点：

| IP 地址 | 网段 | 角色 | HTTP Etag | 端口特征 |
|---------|------|------|-----------|----------|
| 192.168.20.1 | 有线用户网 | 锐捷核心网关 | `2646488223` | 仅 53/80/443 |
| 192.168.101.1 | 设备管理网 | 疑似网关/汇聚 | `2646488223` | 大量端口开放（与光猫指纹高度相似） |
| 10.5.190.1 | 无线用户网 | 无线网段网关 | `2646488223` | 大量端口开放（与光猫指纹高度相似） |
| 192.168.1.1 | 光猫段 | 电信智能网关 | `298-1bc-636c5ed1` | 光猫标准端口集 |

**关键发现：**
- `192.168.101.1` 与 `10.5.190.1` 的 **HTTP 响应头与 192.168.20.1 完全一致**（Etag、Last-Modified、Content-Length 均相同），说明这三个 IP 的 80 端口服务由同一台设备（锐捷网关）提供，或存在反向代理/端口映射。
- 但这两个 IP 的 **TCP 端口扫描结果又与 192.168.1.1（光猫）高度相似**，开放了大量非锐捷特征端口（如 3306、3389、5900 等）。
- **推断：** 学校出口架构可能是"锐捷网关 + 光猫"的双层结构，锐捷网关（192.168.20.1）负责校内三层交换，光猫（192.168.1.1）承担出口 NAT 和透明代理。`192.168.101.1` 和 `10.5.190.1` 可能是光猫在管理网和无线网中的映射地址，或学校出口安全设备对扫描探测进行了统一响应。

### A.4 10.8.7.221 设备分析

该地址出现在 `icube-api.bytedance.net` 的 DNS 解析结果中，疑为学校内部缓存/代理节点：

- **ICMP：** 完全丢弃（`ping` 100% 丢包）。
- **TCP 端口指纹：** 开放 21/22/23/25/53/80/110/111/135/139/143/443/445/993/995/1723/3306/3389/5900/8080，**与 192.168.1.1（光猫）的端口指纹完全一致**。
- **HTTP：** `curl` 返回空内容，与光猫响应不同，可能存在端口转发或虚拟主机区分。

**推断：** `10.8.7.221` 极有可能是学校出口设备（或光猫）在内部网络中的另一张面孔/映射地址，用于将特定域名流量牵引到本地代理/缓存层。

### A.5 多层 NAT 与内网路由异常

- **无线用户**分配 `10.x` 地址，而**有线测试机**在 `192.168.20.0/24`，两者需经过学校核心网关做 NAT 转换。
- 当使用 `mihomo` 等 TUN 模式代理工具时，学校内网 DNS 返回的 `10.x` 地址会被 TUN 路由策略错误转发，导致**内网直连超时**。
- 已验证：解析 `www.baidu.com` 曾得到 `103.235.46.115`，该 IP 在当前网络下直接不可达（100% 丢包），说明学校 DNS/透明代理对特定域名做了**非标准解析或黑洞处理**。
- 进一步验证：通过 Cloudflare DoH 解析 `www.baidu.com` 得到 `103.235.46.102`，TCP 80/443 可通，但 ICMP 丢包率高达 75%，延迟约 222ms。这表明学校透明代理/出口设备对 ICMP 进行了严格过滤或限速，而 TCP 流量被透明代理正常转发。

### 6. 透明代理指纹再确认

在对光猫进行 HTTP 探测时，发现一个**关键证据**：

```http
HTTP/1.1 200 OK
Content-Length: 444
Proxy-Connection: keep-alive
X-Frame-Options: SAMEORIGIN
```

**`Proxy-Connection: keep-alive` 出现在 192.168.1.1 的响应中。**

进一步对比测试：

| 目标 | Proxy-Connection 头 | 分析 |
|------|---------------------|------|
| 192.168.1.1（光猫） | ✅ 存在 | 内网到光猫的流量也被代理 |
| 192.168.20.1（锐捷网关） | ✅ 存在 | 网关自身响应也被代理 |
| www.baidu.com | ✅ 存在 | 外网流量被代理 |

**结论：** 该 `Proxy-Connection` 头**不是光猫自身插入的**，而是位于本机与光猫之间的**透明代理设备**统一添加的。结合网络拓扑，最可能的代理设备是 **192.168.20.1 锐捷网关**，它对所有经过的 HTTP 流量（无论内网还是外网）都做了一层代理转发。

这意味着：
1. 学校网络的所有 HTTP 明文流量**全部可被该设备读取和记录**
2. 该设备有能力在 HTTP 层进行**内容过滤、广告注入、URL 重定向**
3. 如果该设备被攻破，攻击者可获得全校师生的**完整上网行为日志**

  - 这不符合正常 TCP/IP 栈行为，表明目标设备在网络层存在**全端口开放欺骗**或**TCP 连接接受后静默丢弃**的机制（可能是防火墙/IPS 反扫描策略）。

#### 9.5 8080/5555 独立管理平面

- **8080 端口**运行的是不同于 80 端口 LuCI 的服务：
  - `Server: HomeGateway`
  - 返回的登录页与 80 端口的 LuCI 界面**不同**
  - 可能存在独立的认证机制或漏洞入口

- **5555 端口**也返回登录页（大小 15008），与 80/8080 页面均不同。

- **推测：** 该光猫可能同时运行了多套管理平面：
  - `80`：运营商定制的 OpenWrt LuCI（面向用户）
  - `8080`：`HomeGateway` 管理平面（可能面向运营商运维）

  1. TR-069 仅接受来自特定 ACS 服务器 IP 的连接
  2. 该功能在固件中被禁用但端口仍开放
  3. 需要特定的 HTTP 认证/加密握手

3. **密码强度足够:** 500+弱密码均失败，说明用户设置了较强密码
4. **端口隔离:** 路由器80端口被封，管理流量不经过有线网段

### M.4 后续攻击建议

| 方案 | 可行性 | 预期时间 |
|------|--------|----------|
| rockyou.txt字典+握手包离线破解 | 需先捕获握手包 | 1-2小时 |
| 固件逆向分析 | 可能硬编码密码 | 数小时 |
| 物理靠近摄像头侧用高增益天线 | 需现场操作 | 30分钟 |
