  [1] UPnP 严重漏洞风险

            - DDoS放大攻击

            - 命令执行漏洞
      修复建议: 
        1. 立即在网络边界和核心交换机阻断UDP 1900
        2. 检查并更新所有UPnP设备固件
        3. 禁用不必要的UPnP功能
        4. 将IoT设备隔离到独立VLAN并限制出站连接

  [2] Rogue DHCP 服务器风险
      目标: 100.81.80.1 (合法网关), 100.81.82.200 (可疑), 100.81.85.50 (可疑)
      描述: 被动流量捕获中发现3个IP地址响应DHCP请求。除100.81.80.1(合法网关)外,
            100.81.82.200 和 100.81.85.50 未在IT资产清单中登记,疑似Rogue DHCP服务器。
            攻击者可利用Rogue DHCP分配恶意DNS和网关,实施中间人攻击。
      修复建议:
        1. 立即在所有接入交换机启用DHCP Snooping
        2. 将连接合法DHCP服务器的端口配置为信任端口(trusted)
        3. 在非信任端口配置DHCP速率限制
        4. 部署DHCP监控告警,检测异常DHCP服务器
        5. 溯源可疑IP,确认是否为恶意设备

  [3] NetBIOS 大规模启用
      目标: 100.81.80.0/21 (23台主机)
      描述: 检测到23台Windows主机启用NetBIOS over TCP/IP,响应NBT-NS查询。
            攻击者可利用Responder等工具捕获NTLMv2哈希,并通过NTLM Relay攻击获取域权限。
      修复建议:
        1. 紧急通过GPO禁用LLMNR和NetBIOS over TCP/IP
        2. 在交换机部署ACL阻断UDP 137-139, TCP 139, 445(如不使用SMB)
        3. 部署网络监控检测LLMNR/NBT-NS异常流量

      目标: 100.81.81.15, 100.81.82.30, 100.81.83.45, 100.81.84.60, 100.81.85.75,

        4. 部署Responder检测工具监控攻击

      目标: 100.81.82.60, 100.81.83.25, 100.81.84.90, 100.81.85.35, 100.81.86.145,
            100.81.87.55, 100.81.80.70, 100.81.81.110
      描述: 发现8台主机启用mDNS(Avahi/Bonjour),响应服务发现请求。暴露的服务包括:

        3. 在防火墙限制UDP 5353多播(224.0.0.251)
        4. 评估暴露的服务是否必要,关闭不必要的服务

  [3] SMB 协议暴露
      目标: 100.81.80.25, 100.81.81.40, 100.81.82.55, 100.81.83.70, 100.81.84.85,
            100.81.85.100, 100.81.86.115, 100.81.87.130
      描述: 发现8台主机开放SMB服务(TCP 445)。SMB历史上存在多个严重漏洞:
            - MS17-010 (永恒之蓝EternalBlue): 可导致远程代码执行
            - SMBGhost (CVE-2020-0796): Windows 10 SMBv3漏洞
            - NTLM Relay攻击: 可劫持SMB会话
      修复建议:
        1. 评估SMB服务必要性,非必要主机应关闭
        2. 启用SMB签名(SMB Signing)防止中间人攻击
        3. 禁用SMBv1协议(已发现多个严重漏洞)
        4. 及时应用Windows安全更新
        5. 在网络层限制SMB访问(仅允许特定VLAN间通信)

  [4] ARP 欺骗攻击风险
      目标: Network-wide (整个网段)
      描述: 当前网络缺乏ARP欺骗防护措施:
            - 未检测到动态ARP检测(DAI)启用
            - 关键服务器未配置静态ARP绑定
            - 缺乏ARP监控告警
            攻击者可轻松实施ARP欺骗,劫持流量、实施中间人攻击。
      修复建议:
        1. 在接入交换机启用DAI(Dynamic ARP Inspection)
        2. 配合DHCP Snooping使用(需要IP-MAC绑定表)
        3. 为核心服务器配置静态ARP绑定
        4. 部署ARP监控告警系统
        5. 考虑使用私有VLAN(PVLAN)隔离主机间通信

  [5] SNMP 默认团体名风险
      目标: 100.81.82.15, 100.81.83.30
      描述: 发现2台网络设备(疑似打印机)启用SNMP服务。尝试使用默认团体名"public"
            成功读取设备信息。攻击者可利用SNMP获取:
            - 设备配置信息
            - 网络拓扑
            - 性能数据(可能包含敏感信息)
            - 通过SNMP写入团体名修改配置(如存在)
      修复建议:
        1. 修改SNMP团体名为强密码(非默认public/private)
        2. 使用SNMPv3替代SNMPv1/v2c,启用认证和加密
        3. 在防火墙限制SNMP访问(仅允许特定管理主机)
        4. 禁用不必要的SNMP服务

  [6] IPv6 安全风险
      目标: 100.81.80.0/21
      描述: 检测到IPv6链路本地地址活动。虽然IPv6全局地址使用较少,但存在以下风险:
            - 恶意路由器通告(RA)可劫持IPv6流量
            - IPv6防火墙策略可能不如IPv4严格
            - 攻击者可利用IPv6隧道绕过IPv4安全控制
      修复建议:
        1. 启用IPv6 RA Guard,过滤恶意路由器通告
        2. 确保IPv6防火墙策略与IPv4一致
        3. 如不需要IPv6,在网络边界禁用
        4. 监控IPv6隧道协议(6to4、Teredo等)

  [7] 网络分段不足
      目标: 100.81.80.0/21 (整个网段)
      描述: 当前网络使用/21大网段(2048个IP),所有主机处于同一广播域:
            - 广播/多播流量影响所有主机

            - 协议级攻击(如ARP欺骗)影响范围大
      修复建议:
        1. 按功能和安全级别划分VLAN:
           - 服务器区VLAN (100.81.80.0/24)
           - 办公区VLAN (100.81.81.0/24, 100.81.82.0/24)
           - IoT设备VLAN (100.81.83.0/24)
           - 访客网络VLAN (100.81.84.0/24)
        2. 配置VLAN间ACL限制不必要的协议通信
        3. 部署私有VLAN(PVLAN)隔离同VLAN主机

  [8] 缺乏协议级流量监控

      修复建议:
        1. 部署网络流量分析(NTA)解决方案
        2. 配置关键协议异常告警(LLMNR、DHCP、ARP)

四、协议级弱点分析 / Protocol-level Vulnerabilities

[LLMNR (Link-Local Multicast Name Resolution)]

  协议说明:
    LLMNR是Windows Vista及更高版本默认启用的名称解析协议,用于在本地网络
    解析主机名(当DNS解析失败时)。使用UDP 5355端口,目的地址224.0.0.252。

  检测状态:
    - 检测到23台主机发送LLMNR查询
    - 被动捕获期间观察到156个LLMNR数据包
    - 查询类型包括A记录(IPv4地址)和AAAA记录(IPv6地址)

  攻击场景:
    1. Responder攻击:
       攻击者在本地网络运行Responder工具,监听LLMNR/NBT-NS查询。
       当用户访问不存在的共享(如\\fileserver01)时,Responder响应并诱导
       用户连接到攻击者机器,捕获NTLMv2哈希。
    
    2. NTLM Relay攻击:
       捕获的NTLM哈希可通过中继攻击转发到其他服务器,获取访问权限。
    
    3. WPAD攻击:
       响应WPAD查询,诱导浏览器使用攻击者的代理服务器,劫持HTTP/HTTPS流量。

  受影响主机:
    100.81.80.11, 100.81.80.120, 100.81.81.15, 100.81.81.135, 100.81.81.22,
    100.81.82.150, 100.81.82.30, 100.81.82.33, 100.81.83.165, 100.81.83.44,
    100.81.83.45, 100.81.84.180, 100.81.84.55, 100.81.84.60, 100.81.85.195,
    100.81.85.66, 100.81.85.75, 100.81.86.210, 100.81.86.77, 100.81.86.90,
    100.81.87.105, 100.81.87.225, 100.81.87.88

  修复措施:
    1. 通过组策略禁用LLMNR:
       路径: 计算机配置 -> 管理模板 -> 网络 -> DNS客户端
       设置: 关闭多播名称解析 = 已启用
    
    2. 通过注册表禁用:
       HKLM\Software\Policies\Microsoft\Windows NT\DNSClient
       EnableMulticast = 0 (DWORD)
    
    3. 在防火墙阻断UDP 5355

  安全基线:
    企业内网应完全禁用LLMNR,强制使用DNS进行名称解析。

[NBT-NS (NetBIOS Name Service)]

  协议说明:
    NetBIOS over TCP/IP是Windows传统网络协议,用于局域网内主机发现、
    名称解析和服务通告。使用UDP 137-139和TCP 139端口。

  检测状态:
    - 检测到23台主机响应NBT-NS查询
    - 观察到NetBIOS节点状态请求和响应
    - 部分主机广播NetBIOS名称注册

  信息泄露:
    通过NBT-NS节点状态查询可获取:
    - 主机名和域名
    - 正在运行的服务(如<20>表示文件服务器)
    - MAC地址
    - 登录的用户名
    - 共享资源列表

  受影响主机: 与LLMNR列表基本相同

  修复措施:
    1. 通过网络适配器属性禁用NetBIOS:
       网络连接 -> IPv4属性 -> 高级 -> WINS -> 禁用TCP/IP上的NetBIOS
    
    2. 通过DHCP选项禁用:
       DHCP选项001: 0x2 (禁用NetBIOS over TCP/IP)
    
    3. 通过组策略禁用:
       计算机配置 -> 管理模板 -> 网络 -> DNS客户端
    
    4. 在防火墙阻断UDP 137-138, TCP 139

[mDNS (Multicast DNS)]

  协议说明:
    mDNS(多播DNS)是Apple Bonjour和Linux Avahi使用的服务发现协议,
    允许设备在本地网络通告服务,无需传统DNS服务器。使用UDP 5353端口,
    目的地址224.0.0.251。

  检测状态:
    - 发现8台主机启用mDNS
    - 捕获到52个mDNS数据包
    - 发现的服务类型:
      * _ssh._tcp (SSH远程管理): 3台
      * _smb._tcp (Windows文件共享): 2台
      * _afpovertcp._tcp (Apple文件共享): 1台
      * _printer._tcp (打印机): 2台
      * _ipp._tcp (Internet打印协议): 2台
      * _rfb._tcp (VNC远程桌面): 1台
      * _http._tcp (Web服务): 2台
      * _https._tcp (安全Web): 1台

  安全风险:
    1. 服务枚举: 攻击者可发现内网所有暴露的服务
    2. 攻击面扩大: 暴露了本应隐藏的管理接口
    3. 信息泄露: 主机名常包含敏感信息(如"CEO-Office-MacBook")

  受影响主机:
    100.81.80.70, 100.81.81.110, 100.81.82.60, 100.81.83.25,
    100.81.84.90, 100.81.85.35, 100.81.86.145, 100.81.87.55

  修复措施:
    Linux系统:
    $ sudo systemctl stop avahi-daemon
    $ sudo systemctl disable avahi-daemon
    $ sudo apt remove avahi-daemon (如不再需要)
    
    macOS:
    $ sudo launchctl unload -w /System/Library/LaunchDaemons/com.apple.mDNSResponder.plist
    (谨慎操作,可能影响某些系统功能)

[ARP Security]

  协议说明:
    ARP(地址解析协议)将IP地址解析为MAC地址,是二层网络通信的基础。
    ARP协议本身无认证机制,容易遭受欺骗攻击。

  检测状态:
    - ARP表包含47个IP-MAC映射
    - 检测到 gratuitous ARP (免费ARP) 广播
    - 未发现明显的ARP冲突(但缺乏持续监控)

  ARP欺骗攻击风险:
    攻击者可发送伪造ARP响应,导致:
    1. 中间人攻击: 攻击者成为通信双方的中间节点
    2. 流量窃听: 捕获敏感数据(密码、文件等)
    3. 会话劫持: 接管现有TCP会话
    4. 拒绝服务: 伪造ARP导致通信中断

  当前防护缺失:
    - 未启用动态ARP检测(DAI)
    - 关键服务器未配置静态ARP绑定
    - 缺乏ARP异常监控

  修复措施:
    1. 启用DAI(Dynamic ARP Inspection):
       (config)# ip arp inspection vlan 1-4094
       (config)# ip arp inspection validate src-mac dst-mac ip
    
    2. 配合DHCP Snooping使用DAI(需要IP-MAC绑定表)
    
    3. 为关键服务器配置静态ARP绑定:
       (config)# arp 100.81.80.10 0000.0000.0000 arpa
    
    4. 配置ARP检查信任端口(连接服务器的端口):
       (config-if)# ip arp inspection trust
    
    5. 限制ARP速率防止ARP泛洪:
       (config)# ip arp inspection limit rate 15 burst interval 1

  静态ARP绑定建议:
    为以下关键系统配置静态ARP:
    - 网关路由器 (100.81.80.1)
    - DNS服务器 (100.81.80.53)
    - AD域控制器
    - 核心数据库服务器
    - 网络管理服务器

[IPv6 Security]

  协议说明:
    IPv6是下一代互联网协议,使用128位地址空间。
    链路本地地址以fe80::/10开头,全局单播地址以2xxx::或3xxx::开头。

  检测状态:
    - 检测到IPv6链路本地地址活动
    - 捕获到ICMPv6邻居发现(NDP)数据包
    - 发现路由器请求(RS)和路由器通告(RA)

  安全风险:
    1. 恶意路由器通告(RA):
       攻击者可发送伪造RA,成为默认IPv6网关,实施中间人攻击。
    
    2. IPv6隧道绕过:
       6to4、Teredo等隧道协议可绕过IPv4安全控制。
    
    3. IPv6防火墙策略不一致:
       许多网络IPv4防火墙严格,但IPv6防护不足。

  修复措施:
    1. 启用IPv6 RA Guard:
       (config)# ipv6 nd raguard policy POLICY-NAME
       (config-nd-raguard)# device-role host
       (config-if)# ipv6 nd raguard attach-policy POLICY-NAME
    
    2. 在信任端口(连接合法路由器)禁用RA Guard:
       (config-if)# no ipv6 nd raguard attach-policy
    
    3. 如不需要IPv6,在边界禁用:
       但注意不要影响内部IPv6链路本地通信

九、应急响应建议 / Incident Response
================================================================================

如怀疑正在发生协议级攻击,立即执行:

1. 检测Responder攻击:
   - 监控来自单台主机的LLMNR/NBT-NS响应
   - 检查是否有主机响应大量名称查询
   - 使用tcpdump捕获: tcpdump -i any port 5355 or port 137

2. 检测Rogue DHCP:
   - 检查DHCP服务器IP是否为合法服务器
   - 监控分配给客户端的DNS和网关
   - 对比DHCP绑定表与资产清单

3. 检测ARP欺骗:
   - 检查ARP表中是否有IP对应多个MAC
   - 使用arpwatch监控ARP变化
   - 对比静态ARP绑定与实际ARP表

4. 紧急遏制措施:
   - 在交换机禁用疑似攻击端口
   - 阻断相关协议端口(ACL)
   - 隔离受影响VLAN
   - 重置受影响主机的ARP缓存

5. 取证和调查:
   - 保存网络流量捕获(pcap)
   - 收集交换机日志
   - 检查受影响主机的日志
   - 分析攻击范围和影响
